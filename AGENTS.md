# Agent Instructions

This repository is for `fel-dolby-vision-movies`: a Python automation pipeline
that discovers physical-media forum sources, validates Dolby Vision Profile 7
FEL Blu-ray releases, normalizes release metadata, and publishes
machine-readable data plus GitHub Pages artifacts.

## Superpowers System

Superpowers skills are installed through Codex native skill discovery:

```text
~/.agents/skills/superpowers -> ~/.codex/superpowers/skills
```

Codex discovers these skills at startup. Restart Codex after installing or
updating the symlink.

## Operating Rules

- Read the current files before acting. `CONTEXT.md` is ignored local
  historical context and may be absent or stale; use it only as background, not
  as a tracked source of truth.
- Use superpowers brainstorming before changing product behavior or widening
  scope.
- Check `git status --short --branch` before editing in this repository.
- Keep `.env`, API keys, tokens, cookies, cached HTML, and local credentials
  out of git.
- Do not print secrets back into chat or commit logs.
- Treat source discovery and forum scraping as brittle network behavior:
  prefer deterministic tests with mocked HTML before relying on live sites.
- Separate verified FEL evidence from inferred Dolby Vision or generic REMUX
  evidence.
- Do not list a movie unless the parser can establish direct one-to-one
  correlation between the specific release and Profile 7 FEL.
- Prefer `just` recipes for local work because they set `PYTHONPATH=src`.
  Running `uv run python -m main ...` directly can fail unless the same path is
  configured.

## Repository Shape

- This is already bootstrapped as a Python project. Keep modules flat directly
  under `src/`; do not reintroduce a nested package directory or root-level
  utility scripts.
- The local command surface is `just --list`, `just lint`, `just test`,
  `just ci`, `just search-for-sources`, `just scrape-for-titles`, `just run`,
  `just compare-found`, and `just migrate`.
- `src/main.py` also exposes `ai-scrape`. There is not currently a `just`
  recipe for it, so run it with `PYTHONPATH=src` or from CI's existing command
  shape when needed.
- `data/forums.txt` is the forum/list source registry used by `--sources`.
  `data/google_sheets.txt` is the supplemental Google Sheets registry that the
  scraper loads automatically when the source path is `data/forums.txt`.
- `FEL.txt` and `raw_fel.txt`, when present locally, are migration inputs for
  `just migrate`; they are not the published database.

## Pipeline Requirements

- Maintain `data/forums.txt` and `data/google_sheets.txt` as committed source
  registries. Automatic discovery may update `data/forums.txt`; do not broaden
  discovery filters without mocked tests for accepted and rejected URLs.
- Use optional Brave Search discovery when `BRAVE_SEARCH_API_KEY` is available.
- Use `FORUM_COOKIE_HEADER` only from local or CI environment when a source
  requires cookies; never commit cookies or fetched HTML.
- Normalize parser/free-text audio into stable values such as `TrueHD Atmos`,
  `DD+ Atmos`, `TrueHD`, `DTS-HD MA`, `DD+`, and `DTS:X`. Blu-ray.com
  enrichment currently preserves richer disc-track labels such as
  `Dolby TrueHD/Atmos 7.1`; do not collapse those labels without tests and an
  explicit public-output decision.
- Generate machine-readable release data in `data/releases.json` and a static
  GitHub Pages dashboard in `dist/`. `README.md` is a hand-maintained static
  file and is not generated by the pipeline.
- `data/posters/` contains committed poster assets. `dist/index.html`,
  `dist/releases.json`, and `dist/posters/` are generated deployment artifacts;
  `dist/` stays ignored locally.
- Preserve enrichment fields in release data: `tmdb_id`, `media_type`,
  `imdb_id`, `poster_path`, `release_url`, `bluray_url`,
  `bluray_release_date`, `hdr_formats`, and `audio_languages`.
- `tmdb_id` is only meaningful together with `media_type` ("movie" or
  "tv"): TMDB movie and TV ids are independent numeric sequences, so
  identity comparisons must use the namespaced pair (see
  `FelRelease.tmdb_identity`), never the bare numeric id. Rows without a
  `media_type` predate TV support and load as movies.
- Sort published release tables by release date, newest first.
- Exclude release groups from user-facing outputs unless a later spec
  explicitly adds them.

## AI-Assisted Extraction

- AI-assisted extraction (`ai-scrape`) runs in the daily Pages workflow after
  the deterministic `run` command.
- CI passes `secrets.CODEX_API_KEY` as `OPENAI_API_KEY` for `ai-scrape`.
  Local runs may use `OPENAI_API_KEY` or `CODEX_API_KEY`; tests must not require
  either secret.
- AI-discovered releases must be tagged with `evidence_type="ai-extracted"` and
  `source_label="codex-ai"`.
- `ai-scrape` must merge into existing `data/releases.json`; it must never
  replace deterministic scraper results.
- `compare-found` is the diagnostic path for comparing AI-assisted extraction
  with the deterministic parser. Keep its CSV and text artifacts useful for
  manual review when changing either extraction path.

## Validation

- Write tests before parser behavior where practical.
- Use mocked forum HTML for correlation, false-positive, malformed HTML, audio
  normalization, and artifact-generation tests.
- Target near-complete coverage for deterministic parser and normalization
  logic.
- The benchmark suite should compare parser outcomes against curated expected
  labels.
- For source discovery, Google Sheets, Reddit, TMDB, blu-ray.com, and AI paths,
  prefer mocked network responses in tests. Do not make basic CI depend on live
  external services or optional secrets.
- Run `just ci` before claiming completion unless the user explicitly narrows
  validation. It runs lint, format check, tests with coverage, and the parser
  benchmark.
- Before claiming completion, report the commands run and any validation gaps.

## Git And CI

- Stage only files intentionally changed for the task.
- Prefer an isolated git worktree for feature or documentation branches when
  the main checkout has unrelated local changes.
- Pull or fetch the latest `main` before creating a branch intended for PR.
- The CI workflow runs on pull requests and on pushes to `main` and `feat/**`.
  Branches outside those push patterns still get CI once a PR is opened.
- CI uses Python 3.13, `uv`, `PYTHONPATH=src`, Ruff, pytest with coverage, and
  `python -m benchmark tests/fixtures/benchmark_cases.json`.
- The scheduled scrape workflow runs tests before scraping, runs the
  deterministic scraper, runs `ai-scrape`, and opens or updates one persistent
  `daily-fel-refresh` pull request when new FEL releases are added to
  `data/releases.json`. It must not push refreshed data directly to protected
  `main`.
- The Pages-only workflow publishes `dist/` after data/code changes are merged
  to `main`; it does not scrape or run the AI worker.
- The manual Secret Smoke Test only verifies `BRAVE_SEARCH_API_KEY`; it is not
  required for normal CI.
- Basic CI must not require optional API secrets to pass.

## Trakt List Sync

`.github/workflows/trakt-sync.yml` mirrors `data/releases.json` into the Trakt
list `yellowbrick242/xbmc4lyfe-fel-content`. It runs on push to `main` whenever
`data/releases.json` changes and twice daily at 10:53 and 22:53 UTC as a
safety net.
The sync code lives in `src/trakt_sync.py` and is invoked via
`python -m main trakt-sync` (or `just trakt-sync` / `just trakt-sync-dry`
locally). Required repo secrets: `TRAKT_APP_CLIENT_ID`,
`TRAKT_APP_CLIENT_SECRET`, `TRAKT_REFRESH_TOKEN`, and `TRAKT_SYNC_PAT` (a
fine-grained PAT with `Secrets: Read and write` on this repo, used by the
workflow to rotate `TRAKT_REFRESH_TOKEN` after each successful run). To mint
the initial refresh token, run `just trakt-bootstrap` locally once.

**Don't burn the refresh-token chain.** Every successful refresh consumes the
old token and issues a new one. The workflow persists the new one back to
the secret automatically, but every *local* `just trakt-sync*` or
`just trakt-bootstrap` call also consumes a token without writing it
anywhere CI can see, leaving the secret stale. Trakt also rate-limits
`/oauth/token` aggressively (a stampede can lock the endpoint for ~45
minutes per-account). Practical rules:
- Don't run `just trakt-bootstrap` more than once unless you've verified the
  previous token is genuinely broken.
- Don't run local sync commands while CI is healthy — let the workflow own
  the rotation chain.
- If you hit `429 REFRESH_TOKEN_API_GET_LIMIT`, wait the full Retry-After
  window before any further attempts. The code raises `TraktRateLimitError`
  with `retry_after_seconds` rather than burning more retries.
